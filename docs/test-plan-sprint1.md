# Atlas Sprint 1 测试计划

> 项目：Atlas 智能排班系统
> Sprint：Sprint 1（2026-03-11 ~ 2026-03-17）
> QA：Andy
> 依据文档：`SPRINT1.md`、`README.md`、`docs/api.md`、`docs/database.md`、`docs/frontend-pages.md`

---

## 1. 测试目标

Sprint 1 的目标是完成项目初始化、基础架构搭建、数据库设计以及企微登录流程打通。本轮测试以“基础可用、文档齐备、关键链路可验证”为主，不覆盖完整业务功能测试。

本轮测试重点：
- 验证项目仓库与基础工程可正常运行
- 验证前后端脚手架达到可开发状态
- 验证数据库设计文档是否覆盖核心实体
- 验证企微登录链路具备可跑通条件
- 提前暴露环境配置、依赖缺失、第三方集成等高风险问题

---

## 2. 测试范围

### 2.1 本 Sprint 纳入测试范围
- GitHub 仓库初始化结果
- 前端脚手架启动能力
- 后端脚手架启动能力
- 数据库设计文档完整性
- 企微 OAuth2 登录流程

### 2.2 本 Sprint 暂不纳入测试范围
- 排班 CRUD 详细功能
- 请假同步逻辑
- 人数校验规则准确性
- 运营经理审批流
- 员工端/店长端完整业务交互
- 性能、压测、安全专项测试

## 3. Sprint 1 验收项

以下验收项基于 `SPRINT1.md` 的验收标准，并结合 PRD 做了可测试化拆解。

### 3.1 验收项 A：GitHub 仓库可访问
**目标**：仓库已创建并具备基础说明文档。

**检查点**：
- 仓库地址存在且可访问
- 默认分支可查看
- `README.md` 已提交
- 仓库名称、项目说明与 Atlas 项目一致

**通过标准**：
- QA 可成功访问仓库主页
- 仓库内存在基础项目文件，非空仓库

---

### 3.2 验收项 B：前端 `npm run dev` 正常启动
**目标**：前端工程可本地启动，具备可访问页面。

**检查点**：
- `atlas-web/` 目录存在
- `package.json` 存在且依赖可安装
- 执行 `npm install` 无阻塞性错误
- 执行 `npm run dev` 成功启动
- 本地开发地址可访问
- 页面至少能正常渲染基础框架/占位页

**通过标准**：
- 前端开发服务启动成功
- 浏览器访问返回 200 / 正常页面
- 控制台无阻塞性启动错误

---

### 3.3 验收项 C：后端服务正常启动
**目标**：后端工程可本地启动，具备基础 API 能力。

**检查点**：
- `atlas-server/` 目录存在
- `package.json` 存在且依赖可安装
- 执行 `npm install` 无阻塞性错误
- 执行 `npm start` 成功启动
- 存在基础健康检查接口（建议 `/health` 或 `/api/health`）
- 接口调用有明确响应

**通过标准**：
- 后端服务启动成功
- 至少一个基础 API 可访问并返回预期结构
- 服务日志无明显启动异常

---

### 3.4 验收项 D：数据库表设计文档完成
**目标**：数据库设计满足当前 MVP 与 Sprint 后续开发的最小需要。

**检查点**：
- `docs/database.md` 已存在
- 覆盖以下核心实体：
  - 用户（user）
  - 门店（store）
  - 排班（schedule / shift / schedule_detail）
  - 请假（leave）
- 字段定义清晰，包括主键、外键、关键状态字段
- 能支撑 PRD 中的角色、门店、排班、请假关系
- 能为后续企微用户同步与请假同步预留必要字段

**通过标准**：
- 文档结构清晰、字段可读
- 核心表完整，不缺失关键业务实体
- 表间关系可被研发直接用于建模

---

### 3.5 验收项 E：企微登录流程跑通
**目标**：用户可通过企微 OAuth2 完成登录闭环。

**检查点**：
- 已配置企微应用基础参数（至少测试环境可用）
- 前端存在登录入口或授权跳转逻辑
- 后端存在 OAuth2 回调接口
- 用户授权后可获取用户身份标识（如 userid / code 换取身份）
- 登录成功后可建立本地会话、token 或登录态
- 异常场景可处理：无 code、code 失效、用户取消授权

**通过标准**：
- QA 按标准流程可完成一次成功登录
- 登录后系统可识别当前用户身份
- 失败场景返回明确错误，不出现白屏/500 未处理异常

## 5. 测试方法与环境建议

### 5.1 测试方法
- **文档检查**：用于仓库、README、数据库设计文档验收
- **安装/启动验证**：用于前后端工程可运行性确认
- **接口验证**：用于后端健康检查、OAuth 回调验证
- **手工端到端测试**：用于企微登录流程跑通
- **日志验证**：检查启动日志、错误日志、回调日志

### 5.2 建议测试环境
- Node.js：与开发约定版本一致
- npm：稳定版本
- 浏览器：Chrome 最新稳定版
- 企微：测试企业环境 / 测试应用配置
- 网络：可访问企微相关接口

### 5.3 入口准入条件
建议满足以下条件后开始 Sprint 1 验收测试：
- 前后端代码已提交到同一仓库或清晰目录结构
- `.env.example` 或环境变量说明已提供
- 企微测试账号、测试应用配置可用
- 数据库设计文档已有初稿

### 5.4 退出条件
- 所有 P0 冒烟项通过
- Sprint 1 五项验收项均有结果（通过/阻塞/失败）
- 阻塞问题已登记并分配责任人

## 6. 风险点

### 6.1 第三方依赖风险：企微配置未就绪
**表现**：缺少 corpid、secret、agentid、回调地址白名单等配置，导致 OAuth 无法联调。

**影响**：Sprint 1 最关键的“企微登录跑通”无法完成真实验收。

**建议**：
- 尽早确认测试环境配置
- 提供 mock 登录方案作为开发兜底
- 将真实企微联调与本地假数据联调拆开验收

---

### 6.2 工程初始化不完整风险
**表现**：目录存在但缺少可运行脚本、依赖声明、环境变量说明。

**影响**：验收只能看文件，不能看真实运行结果。

**建议**：
- 前后端都必须提供启动命令
- 增加 `.env.example`
- 增加健康检查接口与启动说明

---

### 6.3 数据库设计覆盖不足风险
**表现**：只设计基础表，没有考虑角色、审批状态、企微映射字段、排班明细粒度。

**影响**：Sprint 2/3 开发时频繁返工，接口和模型不稳定。

**建议**：
- 在 Sprint 1 即预留用户角色、门店归属、企微用户标识、请假审批单号等字段
- 对“排班表”和“排班明细”是否拆表尽早定稿

---

### 6.4 登录成功但系统身份未落库风险
**表现**：企微授权成功，但系统内没有用户映射或登录态持久化。

**影响**：表面登录成功，实际上无法进入后续业务流程。

**建议**：
- 明确登录成功后的最小闭环：换取身份 -> 查询/创建本地用户 -> 建立 session/token
- 至少补一个“当前登录用户信息”接口用于验收

---

### 6.5 环境差异风险
**表现**：开发机可以运行，其他成员机器无法启动。

**影响**：多人协作效率下降，QA 环境复现困难。

**建议**：
- 固定 Node 版本
- 提供统一启动说明
- 尽早加入 lint / 基础 CI 校验

---

## 7. 后续回归测试建议

Sprint 1 结束后，建议把以下内容纳入后续回归测试基线，避免后续功能开发破坏底层能力。

### 7.1 基础能力回归（每个 Sprint 必回归）
- 前端是否仍可正常启动
- 后端是否仍可正常启动
- 健康检查接口是否可用
- 登录页是否可访问
- 企微登录主链路是否可用
- 核心环境变量说明是否更新

### 7.2 登录与权限回归
- 员工身份登录后权限范围正确
- 店长身份登录后可访问店长端页面
- 运营经理身份登录后可访问审批相关页面
- 未登录访问受限页面时会被拦截
- 登录失效/退出后可正确跳回登录页

### 7.3 数据模型回归
- 数据库文档与实际实现保持一致
- 用户、门店、排班、请假等核心表字段无破坏性变更
- 新增字段不会影响既有接口或登录逻辑

### 7.4 为 Sprint 2/3 提前准备的回归场景
- 排班表创建后基础展示正常
- 请假数据同步不会影响登录与用户识别
- 人数校验逻辑新增后，不影响既有排班保存流程
- 特殊排班审批链路加入后，不影响普通排班提交

### 7.5 自动化建议
建议从 Sprint 1 结束后开始逐步补充以下自动化：
- 前端基础构建/启动检查
- 后端健康检查接口自动化
- OAuth 关键逻辑的单元测试（code 校验、异常处理、用户映射）
- 关键 API 的 smoke test
- PR 合并前最小 CI 校验（安装、构建、测试）

## 9. 结论

Sprint 1 的 QA 核心不是验证业务完整性，而是验证 Atlas 项目是否已经具备“能继续开发”的地基。只要仓库、前后端脚手架、数据库设计、企微登录链路四件事站稳，Sprint 2 才能顺利进入排班功能开发。
