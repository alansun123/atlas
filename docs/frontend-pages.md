# Atlas 前端页面设计（MVP）

> 项目：Atlas 企微门店智能排班系统  
> 角色：Frontend Dev（Cock）  
> 版本：v1.0  
> 更新日期：2026-03-11

---

## 1. 文档目标

基于现有 PRD、`docs/architecture.md` 与 `atlas-web/` 目录骨架，明确 Atlas H5 前端在 MVP 阶段的页面范围、字段与交互、路由结构、页面流转，以及 Day1-Day4 的前端交付边界。

这份文档只覆盖 **MVP 首期必需页面**，坚持一个原则：

**前端先把“登录 → 进入首页 → 看班表 / 做排班 / 审批特殊排班”这条主链路跑通，不做过早复杂化。**

---

## 2. 设计前提

### 2.1 产品前提

来自 PRD 的核心范围：

- 员工：查看个人班表、提交请假（跳企微）、换班申请（P1）
- 店长：创建/编辑/发布排班、校验人数规则、员工管理
- 运营经理：审批特殊排班、查看排班（抄送）
- 企微为身份源，请假以企微审批结果为准

### 2.2 技术前提

基于当前脚手架与架构文档，前端采用：

- Vue 3 + Vite
- Vue Router
- Vant
- 后续接入 Pinia / API 请求层 / 路由守卫
- 运行形态为 **企微内 H5**，优先移动端体验

### 2.3 页面设计原则

1. **移动优先**：面向企微内 H5，默认单栏布局。
2. **角色收敛**：同一套路由框架下按角色展示不同入口。
3. **列表优先于复杂画布**：MVP 用卡片、列表、分组表格实现排班，不做复杂拖拽排班盘。
4. **规则结果可见**：每次排班校验必须让店长看见“哪里有问题、为什么要审批”。
5. **审批闭环清晰**：特殊排班必须能从校验结果直接进入审批提交，再由运营经理审批。

## 3. MVP 页面清单

以下为建议的 MVP 页面清单。

| 登录回调页 | `/auth/callback` | 全部 | P0 | 处理企微 code、建立 session、跳转首页 |

| 首页（企微登录后） | `/home` | 全部 | P0 | 登录后统一落点，按角色展示工作台 |
| 员工班表页 | `/employee/schedule` | 员工 | P0 | 查看个人周班表 / 今日班次 / 请假冲突提示 |
| 店长排班页 | `/manager/schedule` | 店长 | P0 | 周排班草稿、编辑、校验、提交审批、发布 |

### 3.1 MVP 必做页面（建议严格控制）

P0 先做 8 个页面：

1. `/login`
2. `/auth/callback`
3. `/pending-access`
4. `/home`
5. `/employee/schedule`
6. `/manager/schedule`
7. `/approvals`
8. `/approvals/:id`

如果开发时间更紧，`/manager/schedule/:batchId` 可以先并入 `/manager/schedule` 作为抽屉/二级子视图处理，但从信息架构上我仍建议保留独立详情页。

## 5. 登录引导页 `/login`

### 5.1 页面目标

- 作为企微登录的统一入口
- 检测是否处于企微环境
- 提示用户进入授权流程

### 5.2 页面内容

**页面字段/元素**

- Logo / 产品名：Atlas
- 简短说明：门店智能排班系统
- 当前环境提示
  - 在企微内：可直接登录
  - 不在企微内：提示“请在企业微信中打开”
- 主按钮：`企微授权登录`
- 次按钮：`重试` / `复制测试地址`（仅测试环境）

### 5.3 核心交互

- 页面加载时检测企微 UA / JS-SDK 环境
- 若已有 token，直接跳转 `/home`
- 点击 `企微授权登录`：
  - 调用后端 `/api/auth/wework/url` 或按约定直接拼 OAuth URL
  - 跳转企微授权
- 若非企微环境：
  - 不允许正常登录
  - 显示说明：请从企业微信工作台进入 Atlas

### 5.4 异常状态

- 网络错误：展示“网络异常，请稍后重试”
- 授权地址获取失败：展示 toast + 重试按钮
- 浏览器打开：展示受限提示

## 6. 登录回调页 `/auth/callback`

### 6.1 页面目标

- 接收企微回跳的 `code`
- 调用后端换取用户身份与 JWT
- 根据角色进入不同首页

### 6.2 页面内容

**页面字段/元素**

- loading 图标
- 文案：`正在登录，请稍候`
- 错误态文案
- 重试按钮

### 6.3 核心交互

- 读取 URL 参数：`code`、`state`
- 调用 `POST /api/auth/wework/callback`
- 成功后保存 token / 当前用户信息
- 根据角色跳转：
  - employee → `/home`
  - manager → `/home`
  - operation → `/home`
- 若后端返回“未开通角色” → `/pending-access`

### 6.4 异常状态

- 缺少 code：展示登录失败并可返回 `/login`
- callback 接口失败：展示错误信息 + 重试
- token 写入失败：清空状态并返回登录页

## 7. 待开通页 `/pending-access`

### 7.1 页面目标

- 用户已通过企微识别，但本地未分配角色/门店时承接访问

### 7.2 页面内容

- 图标：待开通 / 锁定
- 标题：`账号待开通`
- 说明文案：
  - 已识别企业微信身份
  - 当前未配置系统角色或门店权限
- 联系提示：请联系管理员开通
- 按钮：`刷新状态`

### 7.3 核心交互

- 点击 `刷新状态`：重新拉取 `/api/auth/me`
- 若角色已配置，跳转 `/home`

## 8. 首页（企微登录后）`/home`

### 8.1 页面目标

这是登录后的统一首页，也是 MVP 最关键的页面之一。

它不应该只是个空白工作台，而是一个 **角色分流 + 今日重点信息汇总页**。

### 8.2 页面结构

建议首页分成 4 个区块：

1. 顶部用户信息区
2. 今日摘要区
3. 快捷入口区
4. 待办提醒区

### 8.3 页面字段

#### 8.3.1 顶部用户信息区

- 用户姓名
- 角色标签（员工 / 店长 / 运营经理）
- 所属门店
- 今日日期 / 星期

#### 8.3.2 今日摘要区

根据角色展示不同摘要：

**员工**
- 今日是否有班
- 今日班次时间
- 本周排班天数
- 是否存在请假冲突/换班处理结果

**店长**
- 当前门店
- 本周排班状态（草稿 / 待审批 / 已发布）
- 今日请假人数
- 待处理异常数（人数不足、超编、请假冲突等）

**运营经理**
- 待审批特殊排班数量
- 今日待关注门店数
- 最近已审批记录数

#### 8.3.3 快捷入口区

**员工**
- `查看我的班表`
- `去企微提交请假`
- `查看审批/消息`（后续可扩）

**店长**
- `进入店长排班`
- `查看审批状态`
- `员工管理`（P1）

**运营经理**
- `处理特殊排班审批`
- `查看门店排班`

#### 8.3.4 待办提醒区

- 待审批事项
- 请假冲突提醒
- 排班尚未发布提醒
- 特殊排班被驳回提醒

### 8.4 核心交互

- 首页加载时调用：
  - `/api/auth/me`
  - 角色对应摘要接口（后续拆分）
- 点击快捷入口跳转对应业务页
- 允许下拉刷新首页信息

### 8.5 首页为什么必须做

因为 Atlas 是角色型工具，不是内容型工具。  
如果登录后没有一个清晰首页，用户就会在多个入口之间迷路。MVP 阶段必须让不同角色一进系统就知道自己该做什么。

## 9. 员工班表页 `/employee/schedule`

### 9.1 页面目标

这是员工侧的核心 P0 页面，优先级非常高。

目标：

- 让员工快速查看自己本周/今日班表
- 能看到请假对班表的影响
- 能快速跳转企微请假

### 9.2 页面结构

建议采用：

1. 顶部周切换条
2. 今日班次卡片
3. 本周班表列表
4. 底部快捷操作区

### 9.3 页面字段

#### 9.3.1 顶部周切换条

- 当前周范围：`03/09 - 03/15`
- 上一周 / 下一周
- `回到本周`

#### 9.3.2 今日班次卡片

- 是否有班
- 班次名称
- 上班时间
- 下班时间
- 门店名称
- 状态：正常 / 已调班 / 与请假冲突

#### 9.3.3 本周班表列表

每一天展示：

- 日期
- 星期
- 班次名称
- 时间段
- 门店
- 状态标签：
  - 已发布
  - 调整中
  - 请假冲突
  - 休息

#### 9.3.4 底部快捷操作区

- `去企微请假`
- `刷新班表`
- `申请换班`（P1 置灰或隐藏）

### 9.4 核心交互

- 默认加载本周个人班表：`GET /api/schedules/me?start=&end=`
- 周切换重新拉取数据
- 点击某天卡片可展开详情（班次备注 / 调整说明）
- 点击 `去企微请假`：跳转企微审批链接
- 检测到冲突时，显示醒目提醒条：
  - `你在 3/14 的班次与已批准请假冲突，请联系店长`

### 9.5 空状态

- 本周无班：显示“本周暂无已发布班次”
- 未发布：显示“店长尚未发布本周排班”

## 10. 店长排班页 `/manager/schedule`

### 10.1 页面目标

这是店长端的核心页面，也是 MVP 最复杂的页面。

它至少要完成：

- 选择排班周
- 查看/编辑一周排班草稿
- 对每个班次安排员工
- 发起校验
- 根据校验结果决定：直接发布 or 提交审批

### 10.2 页面结构

建议按移动端可用性拆为 5 个模块：

1. 门店与周选择区
2. 排班状态总览区
3. 每日班次编辑区
4. 校验结果区
5. 底部操作栏

### 10.3 页面字段

#### 10.3.1 门店与周选择区

- 当前门店名称
- 周范围选择
- `复制上周排班`（P1）
- 排班批次状态：草稿 / 待审批 / 已发布

#### 10.3.2 排班状态总览区

- 本周总班次数
- 已安排员工数
- 冲突数
- 待审批异常数
- 今日请假人数

#### 10.3.3 每日班次编辑区

建议以“按天分组”的卡片列表实现，每天下面有多个班次卡片。

**每个班次卡片展示字段：**

- 日期
- 班次名称（早班 / 中班 / 晚班等）
- 时间范围
- 最小人数
- 最大人数
- 当前已排人数
- 已排员工列表
- 员工状态标签：
  - 请假中
  - 新员工首周
  - 重复排班风险

**编辑时可操作字段：**

- 添加员工
- 删除员工
- 备注
- 标记特殊原因（促销/旺季/临时调班等）

#### 10.3.4 校验结果区

这是 MVP 必须重点做的。

按严重程度分组展示：

**阻断型问题**
- 人数低于最小值
- 人数高于最大值
- 员工请假冲突
- 同时段重复排班

**需审批问题**
- 新员工首周排班
- 临时调班（非自愿）
- 加班/减班
- 旺季/促销特殊安排

每条问题至少展示：

- 问题类型
- 日期/班次
- 涉及员工
- 原因说明
- 处理建议

#### 10.3.5 底部操作栏

- `保存草稿`
- `校验排班`
- `提交审批`
- `发布排班`

按钮状态规则：

- 未校验：仅可保存/校验
- 存在阻断问题：不能发布，可继续编辑
- 仅存在需审批问题：可提交审批，不可直接发布
- 无问题：可直接发布

### 10.4 核心交互

#### 10.4.1 创建/编辑排班

- 选择周后，如果已有批次则加载，否则创建新草稿
- 点击班次卡片进入员工选择弹层
- 员工选择列表需展示：
  - 员工姓名
  - 岗位/角色（如后续有）
  - 是否请假
  - 是否新员工

#### 10.4.2 校验排班

- 点击 `校验排班`
- 调用 `POST /api/schedule-batches/:id/validate`
- 返回结果后刷新校验区
- 若命中审批规则，页面显示“需提交审批”引导

#### 10.4.3 提交审批

- 仅当存在例外规则时出现
- 弹出表单：
  - 审批原因摘要（后端可返回默认文本）
  - 店长补充说明（必填）
- 提交后：
  - 批次状态变为 `pending_approval`
  - 跳转审批详情或返回列表提示“已提交”

#### 10.4.4 发布排班

- 无异常时直接发布
- 发布前二次确认
- 发布成功后 toast：`已发布，并通知相关员工`

### 10.5 为什么这页不建议一上来做拖拽

拖拽排班看起来高级，但它会吃掉大量时间在交互细节和移动端适配上。  
MVP 阶段用“按天 + 按班次 + 员工选择弹层”的方式，更稳、更快、更容易联调后端规则。

## 11. 审批页 `/approvals`

### 11.1 页面目标

承接特殊排班审批流。

对运营经理来说，这是 P0 核心页；对店长来说，它也是查看自己提交审批状态的重要页。

### 11.2 页面结构

1. Tab 切换
2. 审批卡片列表
3. 筛选条件

### 11.3 页面字段

#### 11.3.1 Tab 切换

- 待审批
- 已通过
- 已驳回
- 我提交的（店长视角）

#### 11.3.2 筛选条件

- 门店
- 日期范围
- 审批状态

#### 11.3.3 审批卡片字段

- 门店名称
- 提交人（店长）
- 周范围
- 触发原因摘要
- 提交时间
- 当前状态
- 异常条数

### 11.4 核心交互

- 默认进入“待审批”
- 点击卡片进入 `/approvals/:id`
- 支持下拉刷新
- 运营经理可看到待处理数量徽标

## 12. 审批详情页 `/approvals/:id`

### 12.1 页面目标

让运营经理能基于班表事实做审批，而不是只看一句“请审批”。

### 12.2 页面结构

1. 审批状态头部
2. 基本信息区
3. 触发规则区
4. 关联班表摘要区
5. 审批意见区
6. 底部操作栏

### 12.3 页面字段

#### 12.3.1 审批状态头部

- 审批状态：待审批 / 已通过 / 已驳回
- 门店名
- 提交时间

#### 12.3.2 基本信息区

- 提交人
- 周范围
- 审批类型：特殊排班
- 店长备注说明

#### 12.3.3 触发规则区

每条规则命中展示：

- 类型（低于最小人数 / 超编 / 新员工首周 / 临时调班等）
- 日期
- 班次
- 影响员工
- 说明

#### 12.3.4 关联班表摘要区

- 每天班次概览
- 异常班次高亮
- 涉及员工名单

#### 12.3.5 审批意见区

- 审批意见输入框
- 历史审批记录（若有）

#### 12.3.6 底部操作栏

运营经理看到：
- `驳回`
- `通过`

店长看到：
- 只读状态
- 若被驳回可 `返回修改排班`

### 12.4 核心交互

- 调用 `GET /api/approvals/:id`
- 运营经理点击通过：
  - 二次确认
  - 可选填写意见
  - `POST /api/approvals/:id/approve`
- 点击驳回：
  - 审批意见必填
  - `POST /api/approvals/:id/reject`
- 审批完成后，若通过，则对应排班批次可发布/自动生效（取决于后端策略）

## 13. 路由与页面流转

## 13.1 建议路由表

```text
/login                      登录引导页
/auth/callback              企微登录回调页
/pending-access             待开通页
/home                       登录后首页
/employee/schedule          员工班表页
/manager/schedule           店长排班页
/manager/schedule/:batchId  排班批次详情页
/approvals                  审批列表页
/approvals/:id              审批详情页
/me                         我的（P1）
/manager/staff              员工管理（P1）
/operation/schedules        运营查看排班（P1）
```

## 13.2 页面流转图

### 13.2.1 登录主链路

```text
/login
  ↓ 企微授权
/auth/callback
  ├─ 已分配角色 → /home
  └─ 未分配角色 → /pending-access
```

### 13.2.2 员工主链路

```text
/home
  └─ 查看我的班表 → /employee/schedule
                    └─ 去企微请假（外跳企微）
```

### 13.2.3 店长主链路

```text
/home
  └─ 进入店长排班 → /manager/schedule
                    ├─ 保存草稿
                    ├─ 校验排班
                    ├─ 有例外 → 提交审批 → /approvals
                    └─ 无例外 → 发布排班
```

### 13.2.4 运营经理主链路

```text
/home
  └─ 处理特殊排班审批 → /approvals
                        └─ 查看详情 → /approvals/:id
                                      ├─ 通过
                                      └─ 驳回
```

## 13.3 路由守卫建议

前端需要最基础的路由守卫：

1. 未登录只能访问 `/login`、`/auth/callback`
2. 已登录访问 `/login` 自动跳转 `/home`
3. 员工不能进入店长排班页
4. 店长不能做运营经理审批操作（即使能看到自己的审批记录）
5. 未开通用户只允许访问 `/pending-access`

## 14. 页面与后端接口映射建议

| 页面 | 核心接口 |
|---|---|
| `/login` | `GET /api/auth/wework/url` |
| `/auth/callback` | `POST /api/auth/wework/callback` |
| `/home` | `GET /api/auth/me` + 首页摘要接口 |
| `/employee/schedule` | `GET /api/schedules/me?start=&end=` |
| `/manager/schedule` | `GET /api/stores` / `GET /api/stores/:id/shifts` / `POST /api/schedule-batches` / `GET /api/schedule-batches/:id` / `POST /api/schedule-batches/:id/validate` / `POST /api/schedule-batches/:id/publish` / `POST /api/schedule-batches/:id/submit-approval` |
| `/approvals` | `GET /api/approvals/pending` |
| `/approvals/:id` | `GET /api/approvals/:id` / `POST /api/approvals/:id/approve` / `POST /api/approvals/:id/reject` |

### 14.1 首页摘要接口建议

架构文档里还没明确首页接口，但前端实际非常需要。建议后端补一个聚合接口：

- `GET /api/dashboard/home`

返回按角色裁剪的数据：

- 当前用户信息
- 今日摘要
- 待办数量
- 快捷入口配置

这样首页不会自己拼一堆接口，移动端体验会好很多。

## 19. 最终建议

如果只看 MVP，我的建议很明确：

### 19.1 页面优先级排序

1. `/login`
2. `/auth/callback`
3. `/home`
4. `/employee/schedule`
5. `/manager/schedule`
6. `/approvals`
7. `/approvals/:id`
8. `/pending-access`

### 19.2 四个必须做好的页面

题目要求的四个重点页，也确实是 Atlas 前端的 MVP 核心：

1. **企微登录后的首页** `/home`
2. **员工班表页** `/employee/schedule`
3. **店长排班页** `/manager/schedule`
4. **审批页** `/approvals` + `/approvals/:id`

### 19.3 一个关键判断

Atlas 前端首期不是“做一个漂亮的管理后台”，而是：  
**做一个在企微里能高频使用、角色清晰、路径最短的排班 H5。**

所以页面设计上，宁可朴素一点，也要把：

- 登录清楚
- 首页清楚
- 班表清楚
- 校验清楚
- 审批清楚

先做扎实。
