# Atlas 产品需求说明书 V2.0

## 1. 文档信息

- **产品名称**：Atlas
- **版本**：V2.0
- **产品定位**：门店班次与出勤运营系统
- **适用范围**：门店排班、出勤、请假调班加班、审批、工时、人效分析、基础规则配置
- **说明**：本版用于统一 Atlas 的产品边界、模块结构、规则定义与页面范围，为后续技术设计、库表设计、接口设计、Open API / Private API 文档提供上游依据。

---

## 2. 产品定位

Atlas 不再只是“智能排班工具”，而是一套面向门店用工运营的业务系统。

### 2.1 核心定位

Atlas 定义为：

**门店班次与出勤运营系统**

内部也可进一步表述为：

**企微门店用工运营系统**

### 2.2 核心价值

Atlas 要解决的不是单一“排班”问题，而是完整的门店用工闭环问题：

1. 人什么时候来上班
2. 来了之后是否符合门店运营要求
3. 门店的人效是否健康

### 2.3 核心闭环

Atlas 的核心业务闭环为：

**组织配置 → 排班计划 → 出勤执行 → 异常申请 → 审批流转 → 工时沉淀 → 人效分析**

---

## 3. 产品目标

### 3.1 计划层目标

- 明确每家店、每个岗位、每位员工的班次安排
- 保证排班满足门店规则、岗位要求、技能要求与最低配置要求
- 支持排班草稿、校验、审批、发布、执行中调整

### 3.2 执行层目标

- 让计划班次可以落地执行
- 记录实际上下班打卡情况
- 识别迟到、早退、缺卡、缺勤、跨店支援等场景
- 让请假、调班、加班、补卡等例外流程进入系统闭环

### 3.3 管理层目标

- 沉淀门店工时数据
- 支撑门店目标配置与后续人效分析
- 帮助运营与管理层识别高风险门店、异常用工趋势与长期效率问题

## 4. 产品边界

### 4.1 Atlas 当前要做的

- 组织与基础资料
- 排班与班次管理
- 出勤与考勤执行

- 工时与前置核算
- 门店目标配置
- 人效分析

## 5. 一级模块导航

Atlas 建议采用以下一级模块导航：

1. **工作台**
2. **排班**
3. **出勤**
4. **申请**
5. **审批中心**
6. **工时**
7. **分析**
8. **配置中心**

### 5.1 工作台

提供不同角色的首页摘要与待办入口。

### 5.2 排班

负责班次模板、排班批次、班次编辑、校验、提审、发布与执行中调整。

### 5.3 出勤

负责上下班打卡、出勤记录、计划与实际对比、异常识别、在岗状态。

### 5.4 申请

负责请假、调班、加班、补卡、销假等员工与管理侧业务申请。

### 5.5 审批中心

统一承接各种业务审批流，包括排班例外审批、请假审批、调班审批、加班审批、异常打卡审批等。

### 5.6 工时

负责计划工时、实际工时、加班时长、缺勤扣减、工时汇总与导出。

### 5.7 分析

负责人效、出勤、覆盖率、目标达成、异常趋势等分析能力。

### 5.8 配置中心

负责门店、岗位、员工、班次模板、规则、假种、年假、业绩目标、审批流、角色权限等系统底座配置。

## 9. 关键对象关系

## 9.1 组织对象
- Store（门店）
- Position（岗位）
- Employee（员工）
- Role（角色）
- SkillTag（技能标签）

关系：
- 一个门店有多个员工
- 一个员工归属一个主门店，可支持跨店支援
- 一个岗位可绑定多个技能要求
- 一个员工可拥有多个技能标签

## 9.2 排班对象
- ShiftTemplate（班次模板）
- ScheduleBatch（排班批次）
- ScheduleShift（具体班次）
- ShiftChangeLog（班次变更记录）

关系：
- 一个排班批次属于一个门店
- 一个排班批次包含多个具体班次
- 一个具体班次关联一个员工
- 一个具体班次可产生多个变更记录

## 9.3 出勤对象
- AttendanceRecord（打卡记录）
- AttendanceException（出勤异常）

关系：
- 具体班次对应 0~N 个打卡记录
- 打卡记录可触发异常识别与异常处理

## 9.4 申请对象
- LeaveType（假种）
- LeaveRequest（请假申请）
- ShiftSwapRequest（调班申请）
- OvertimeRequest（加班申请）
- AttendanceCorrectionRequest（补卡申请）

关系：
- 请假申请关联假种
- 各类申请会触发审批实例
- 审批通过后回写班表 / 工时 / 余额 / 出勤状态

## 9.5 审批对象
- ApprovalFlowTemplate（审批流模板）
- ApprovalInstance（审批实例）
- ApprovalNode（审批节点）
- ApprovalActionLog（审批动作记录）

关系：
- 业务单据触发审批实例
- 审批实例包含多个审批节点
- 审批节点产生审批动作日志

## 9.6 工时与额度对象
- WorkHourSummary（工时汇总）
- AnnualLeaveBalance（年假余额）
- AnnualLeaveAdjustment（年假调整记录）

关系：
- 每位员工对应一个或多个年假余额账户
- 年假审批通过后扣减余额
- 年假调整记录影响余额结果

## 9.7 经营目标对象
- StoreSalesTarget（门店业绩目标）
- StorePerformanceSnapshot（门店经营快照，后续扩展）

关系：
- 每家门店可以有多个周期目标
- 目标数据与工时、人效分析联动

## 10. 核心规则细化

## 10.1 排班调整规则

### 规则 A：店长调整店员班次
- 店长可在执行中调整**他人班次**
- 调整可直接生效
- 必须记录变更日志
- 可配置是否强制通知员工

### 规则 B：店长调整本人班次
- 店长调整**自己的班次**不能直接生效
- 必须发起“本人调班申请”
- 必须由运营经理审批通过后方可生效

### 规则 C：所有班次变更必须留痕
至少包括：
- 变更人
- 被影响人
- 原班次
- 新班次
- 变更原因
- 变更时间

---

## 10.2 年假规则

### 规则 A：员工年假独立账户
每位员工拥有独立年假账户，包含：
- 应得额度
- 已用额度
- 剩余额度
- 调整记录

### 规则 B：年假与工龄挂钩
- 工龄越长，年假额度越高
- 年假规则需支持按工龄梯度配置
- 支持对个别员工进行人工修正

### 规则 C：年假扣减与回滚
- 年假申请通过后自动扣减余额
- 驳回、撤回、销假时应按规则回滚或修正余额

---

## 10.3 请假类型配置规则

请假类型必须配置化，不允许硬编码。

支持配置项至少包括：
- 假种名称
- 假种编码
- 是否带薪
- 是否需要附件 / 证明
- 是否占用年假额度
- 最小申请时长
- 最大申请时长
- 默认审批链
- 是否允许员工发起

支持的假种示例：
- 事假
- 年假
- 病假
- 婚假
- 丧假
- 其他

---

## 10.4 门店目标配置规则

### 规则 A：目标按门店配置
- 每家店可维护独立业绩目标

### 规则 B：目标按周期配置
支持：
- 月目标
- 周目标
- 日目标（可选）

### 规则 C：目标与分析联动
- 目标用于后续人效分析
- 后续可作为排班优化与人力匹配的参考输入

## 11. 核心流程图（文本版）

## 11.1 排班流程
配置规则
→ 创建排班批次
→ 编辑班次
→ 校验冲突 / 覆盖 / 技能 / 新员工限制
→ 提交审批（如需要）
→ 发布
→ 员工查看班表
→ 执行中调整
→ 留痕 / 通知

## 11.2 店长本人调班流程
店长发起本人调班申请
→ 运营经理审批
→ 审批通过
→ 更新班表
→ 通知相关人员
→ 工时重新计算

## 11.3 请假流程
员工发起请假
→ 选择假种
→ 校验规则 / 余额 / 附件要求
→ 提交审批
→ 审批通过
→ 回写班表与工时
→ 若为年假则扣减余额

## 11.4 出勤流程
员工查看今日班次
→ 上班打卡
→ 下班打卡
→ 系统比对计划班次
→ 识别迟到 / 早退 / 缺卡 / 缺勤
→ 发起补卡 / 异常说明
→ 审批后修正出勤

## 11.5 人效分析流程
门店配置目标
→ 汇总工时与出勤
→ 汇总目标与实际经营结果
→ 计算人效指标
→ 输出门店 / 员工 / 时段分析结果

## 13. 当前结论

Atlas V2.0 的产品定义已经从“智能排班”升级为“门店用工运营系统”，并明确加入以下关键规则：

- 店长调他人班次可直接生效，调本人班次需运营审批
- 年假按员工账户与工龄规则管理
- 假种必须配置化
- 门店业绩目标必须支持按店、按周期配置

该文档作为后续技术设计、领域模型设计、库表设计、接口清单、Open API / Private API 文档的产品上游基线。
